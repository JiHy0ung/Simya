# 낚시 페이지 (FishPage) 구현 요청

## 목표

`src/pages/Fish/FishPage.jsx`에 낚시로 얻을 수 있는 물고기 도감 페이지를 만든다.
기존 레시피(음식) 페이지와 스타일/구조를 통일한다.

## 0. 작업 전 확인사항 (필수)

- 다음 기존 파일들을 먼저 열어서 구조를 파악한다:
  - `src/constants/town/town.js` — 마을별 어획 가능 물고기 정보가 이미 정의되어 있음. 이 파일을 기준 데이터로 사용한다.
  - `src/assets/food/fish` — 물고기 이미지 디렉토리. `fish.js`가 이미 여기서 이미지를 import하고 있음.
- 프로젝트에서 레시피를 보여주는 페이지를 찾아서(예: `src/pages/Food/FoodPage.jsx` 또는 유사 경로) 다음을 그대로 따른다:
  - 페이지 레이아웃 구조 (헤더, 탭, 카드 그리드 등)
  - 탭 컴포넌트 스타일 (전체/카테고리 탭 전환 방식)
  - 카드 컴포넌트 스타일 (등급별 배지 색상, 이미지 박스, 이름/가격 표시 방식)
  - 색상 토큰, 폰트, 간격 등 디자인 시스템
  - 상태 관리 패턴 (useState로 탭 전환하는지 등)
- 동일한 디자인 토큰(CSS 변수 or Tailwind 클래스)을 재사용한다. 새로운 색상이나 스타일을 임의로 추가하지 않는다.
- 등급별 배지 색상은 기존에 등급(일반/희귀/고급/특급/전설) 배지가 다른 곳에 이미 있다면 그 색상을 그대로 사용한다.

## 1. 데이터

- 물고기 데이터는 `src/data/fish.js` (또는 동일 디렉토리의 fish 데이터 파일)의 `fish` 배열을 사용한다.
- 물고기 이미지는 `src/assets/food/fish` 안에 있다. `fish.js`에서 이미 이 경로의 이미지를 import해서 각 항목의 `image` 필드로 쓰고 있으므로 별도 처리는 필요 없으나, 경로가 깨지지 않았는지 확인한다.
- 각 항목 구조:
  ```js
  { name: "구피", price: 300, grade: "일반", size: "소형", image: Guppy }
  ```
- `grade`: "일반" | "희귀" | "고급" | "특급" | "전설"
- `size`: "소형" | "중형" | "대형"
- 이름이 "특별한 "으로 시작하면 같은 물고기의 특별 버전(가격만 다름, 분해 결과는 일반 버전과 동일하게 취급)이다.

## 2. 마을(지역) 정보

- 어획 가능 마을 정보는 **`src/constants/town/town.js`에 이미 정의되어 있다.** 새로 만들지 말고 이 파일의 구조를 확인해서 그대로 사용한다.
- `town.js`를 먼저 열어서 마을별로 어떤 물고기 목록이 들어있는지, 어떤 키/구조로 되어 있는지 확인한 뒤, `fish.js`의 물고기 이름과 매칭해서 마을 정보를 가져오는 헬퍼 함수를 작성한다 (예: `getVillageByFishName(name)`).
- 만약 `town.js`의 마을 데이터 구조가 아래 매핑과 다르거나 누락된 물고기가 있다면, `town.js` 쪽 데이터를 기준으로 삼고 아래 표는 참고용 백업으로만 사용한다.

참고용 마을 매핑 (세레니티 폭포 / 블루나 해안 / 썬브리즈 사막 / 퓨리 화산):

| 마을          | 소형                                              | 중형                                             | 대형                                                 |
| ------------- | ------------------------------------------------- | ------------------------------------------------ | ---------------------------------------------------- |
| 세레니티 폭포 | 구피, 블루길, 송어, 에메랄드 구피, 은하수 열대어  | 잉어, 연어, 크래피, 유령 베일, 화이트 피쉬       | 유럽 잉어, 강꼬치고기, 삼엽충, 철갑상어, 민물 은갈치 |
| 블루나 해안   | 청어, 멸치, 정어리, 고등어, 크리스탈 해마         | 가자미, 도미, 바닷가재, 유령 해파리, 오로라 피쉬 | 가오리, 참치, 청새치, 상어, 심해 달빛 아귀           |
| 썬브리즈 사막 | 틸라피아, 새우, 대구, 황금 망둑어, 유령 숭어      | 민어, 처브, 오징어, 은빛 삼치, 흰수염메기        | 코비아, 개복치, 배스, 데저트 피쉬, 황금 개복치       |
| 퓨리 화산     | 불가사리, 꽃게, 맹독 복어, 화산 날치, 불꽃 황제돔 | 능성어, 퍼치, 화산 문어, 쏠배감펭, 좀비 물고기   | 만새기, 뱀장어, 빨판상어, 전기 가오리, 섀도우 피쉬   |

(이 표는 `town.js`에 데이터가 없거나 불완전할 경우에만 fallback으로 사용. "특별한 X"는 "X"와 동일한 마을로 매핑)

## 3. 분해(해체) 시 생선살 개수 규칙

물고기를 분해하면 등급에 따라 정해진 개수의 "생선살" 아이템이 나온다.

| 등급 | 개수 |
| ---- | ---- |
| 일반 | 1개  |
| 희귀 | 2개  |
| 고급 | 4개  |
| 특급 | 6개  |
| 전설 | 10개 |

생선살 아이템 이름 규칙: `{마을} {사이즈} 생선살`

- 예: 구피(세레니티 폭포, 소형, 일반) → **세레니티 소형 생선살 1개**
- 예: 도미(블루나 해안, 중형, 희귀) → **블루나 중형 생선살 2개**
- 마을 이름에서 뒤의 "폭포/해안/사막/화산" 등 지역 키워드는 생략하고 앞부분만 사용한다 (세레니티 폭포 → 세레니티, 블루나 해안 → 블루나, 썬브리즈 사막 → 썬브리즈, 퓨리 화산 → 퓨리).
- "특별한 X"를 분해해도 동일하게 X의 등급 기준 개수가 나온다 (가격만 다를 뿐 분해 결과는 같음).

이 분해 로직은 컴포넌트 내부 또는 유틸 함수(`getFishDecomposition(fishName)` 등)로 구현해서, 생선 카드/상세 정보에서 호출해 보여준다.

## 4. 페이지 구조

### 탭

- 다른 레시피 페이지와 동일한 탭 UI 컴포넌트/스타일을 재사용한다.
- 탭 목록: **전체 | 세레니티 폭포 | 블루나 해안 | 썬브리즈 사막 | 퓨리 화산**

### "전체" 탭

- 마을별로 섹션을 나눠서 전체 물고기를 표시한다.
- 각 마을 섹션 안에서 다시 소형 / 중형 / 대형으로 구분해서 보여준다.
- 즉 구조: 마을 섹션(4개) → 사이즈 서브섹션(소형/중형/대형) → 물고기 카드 그리드
- 업로드된 도감 이미지의 레이아웃(마을 헤더 → 소형/중형/대형 행 구분 → 카드)을 참고해서 유사하게 구성한다.

### 마을별 탭 (세레니티 폭포 / 블루나 해안 / 썬브리즈 사막 / 퓨리 화산)

- 해당 마을의 물고기만 소형 / 중형 / 대형으로 구분해서 표시한다.

### 물고기 카드 / 상세 정보

각 물고기 카드(또는 클릭 시 펼쳐지는 상세 영역)에는 다음 정보를 표시한다:

- 물고기 이미지, 이름
- 등급 배지 (일반/희귀/고급/특급/전설 — 기존 등급 배지 색상 재사용)
- 가격
- **어획 가능 마을**
- **분해 시 생선살 개수** (예: "세레니티 소형 생선살 1개" 같은 형태로 표시)

기존 레시피 카드가 "재료 보기" 등 펼침/모달 UI를 쓰고 있다면 동일한 패턴으로 분해 정보를 보여준다. 단순 카드 그리드 형태라면 카드 하단에 바로 표시한다.

## 5. 구현 순서 제안

1. 기존 레시피 페이지 코드 확인 (레이아웃, 탭, 카드 컴포넌트, 스타일)
2. 마을 매핑 데이터 파일 생성
3. 분해 개수 계산 유틸 함수 작성
4. `FishPage.jsx`에서 탭 상태 관리 + 마을/사이즈별 그룹핑 로직 작성
5. 물고기 카드 컴포넌트 작성 (또는 기존 레시피 카드 컴포넌트 재사용/확장)
6. 전체 탭 / 마을별 탭 렌더링 연결
7. 반응형 그리드 확인 (기존 레시피 페이지의 브레이크포인트 따름)

## 6. 주의사항

- "특별한 X" 물고기도 도감에 함께 표시하되(별도 등급 배지나 "특별" 마크 표기는 기존 디자인 시스템에 유사 패턴이 있으면 따르고, 없으면 단순 텍스트로 구분), 분해 결과는 X와 동일하게 처리한다.
- 새로운 색상값이나 임의의 디자인 결정을 추가하지 말고, 항상 기존 코드베이스에서 패턴을 찾아 재사용한다.
- 커밋 메시지는 Korean conventional commit 형식 사용 (예: `feat: 낚시 페이지(FishPage) 구현`).
